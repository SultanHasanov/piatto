# Piatto POS: подготовка production

## 1. Supabase

1. Создайте отдельные проекты staging и production.
2. В каждом проекте примените миграцию `supabase/migrations/202607100001_initial_production_schema.sql` через Supabase CLI или SQL Editor.
3. Создайте пользователя сотрудника в Authentication → Users.
4. Скопируйте `supabase/seed.sql`, замените `v_shop_id` и `v_user_id`, затем выполните SQL. UUID магазина должен совпасть с `VITE_SUPABASE_SHOP_ID`.
5. Убедитесь, что Security Advisor не показывает таблиц без RLS.
6. Не используйте `service_role` или secret key в PWA. Во frontend допускается только publishable key.

## 2. Переменные окружения

Скопируйте `.env.example` в `.env.local` для разработки. В production задайте те же `VITE_*` через панель хостинга. Собирайте staging и production с разными Supabase-проектами.

Если Supabase-переменные отсутствуют, приложение запускается в явно обозначенном локальном режиме: удалённая синхронизация и загрузка изображений отключены.

## 3. Перенос с Mokky

1. До обновления убедитесь, что старое приложение завершило синхронизацию с Mokky, затем экспортируйте резервную копию из настроек.
2. Установите новую сборку. При первом запуске содержимое `localStorage` автоматически переносится в IndexedDB.
3. Войдите в Supabase и откройте Настройки → Синхронизация.
4. Нажмите «Перенести локальные данные в Supabase» один раз. Дождитесь нулевой очереди и статуса «Готово».
5. Сверьте категории, товары, остатки, количество оплаченных/возвращённых заказов и выручку со старым экспортом.
6. Сохраните исходный JSON вне планшета и только после сверки прекратите использование Mokky.

Повторный перенос безопасен по UUID: записи обновляются, а `operation_id` предотвращает повторное применение одной операции. Не запускайте полный перенос одновременно с другого устройства.

## 4. Офлайн-режим

- Каталог, настройки, заказы, очередь и текущая корзина хранятся в IndexedDB.
- Приложение запрашивает persistent storage, но пользователь всё равно может удалить данные сайта вручную.
- Не очищайте данные браузера до подтверждения нулевой очереди синхронизации.
- При конфликте серверной версии операция остаётся в очереди, а ошибка показывается в настройках; не удаляйте очередь до разбора.

## 5. Бэкапы

- На Supabase Pro включены ежедневные резервные копии БД. Дополнительно выполняйте внешний `supabase db dump`/`pg_dump` по расписанию.
- Бэкап БД не восстанавливает удалённые файлы Storage. Копируйте bucket `product-images` отдельно.
- Раз в квартал выполняйте тестовое восстановление в отдельный staging-проект и сверяйте отчёты.

## 6. Выкладка

1. Запустите `npm ci`, `npm run lint`, `npm run build`.
2. Разверните `dist/` на HTTPS-домене (Cloudflare Pages, Vercel или Netlify).
3. Проверьте доступ к Supabase из фактической точки продаж и через резервный мобильный интернет.
4. Установите PWA на планшет, войдите, выполните онлайн- и офлайн-заказ, возврат и редактирование.
5. Проверьте обновление service worker: незавершённая корзина должна восстановиться после перезапуска.
6. Настройте Sentry DSN, 2FA администраторов Supabase и уведомления о расходах/лимитах.

## 7. Фискализация

Piatto POS хранит управленческие заказы и не является зарегистрированной ККТ. До приёма реальных платежей в РФ требуется интеграция с зарегистрированной онлайн-кассой/фискальным регистратором и ОФД либо подтверждённое применимое исключение по 54-ФЗ.

## Откат

- Храните предыдущий статический build и JSON-экспорт Mokky до завершения приёмки.
- При проблеме frontend можно откатить без отката БД, если новая миграция не удаляла столбцы.
- Перед восстановлением БД остановите приём заказов, сделайте свежий dump и зафиксируйте последнюю синхронизированную операцию.
